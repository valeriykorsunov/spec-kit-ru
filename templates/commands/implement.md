## Ввод пользователя

```text
$ARGUMENTS
```

Вы **ОБЯЗАНЫ** учесть ввод пользователя перед продолжением (если он не пуст).

## План действий

1. Запустите `{SCRIPT}` из корня репозитория и разберите `FEATURE_DIR` и список `AVAILABLE_DOCS`. Все пути должны быть абсолютными. Для одинарных кавычек в аргументах (например, "I'm Groot") используйте escape-синтаксис: например, 'I'\''m Groot' (или двойные кавычки, если возможно: "I'm Groot").

2. **Проверьте статус чек-листов** (если существует директория `FEATURE_DIR/checklists/`):
   - Сканируйте все файлы чек-листов в директории `checklists/`
   - Для каждого чек-листа подсчитайте:
     - Всего пунктов: Все строки, соответствующие `- [ ]`, `- [X]` или `- [x]`
     - Выполненные пункты: Строки, соответствующие `- [X]` или `- [x]`
     - Невыполненные пункты: Строки, соответствующие `- [ ]`
   - Создайте таблицу статусов:

     ```text
     | Checklist | Total | Completed | Incomplete | Status |
     |-----------|-------|-----------|------------|--------|
     | ux.md     | 12    | 12        | 0          | ✓ PASS |
     | test.md   | 8     | 5         | 3          | ✗ FAIL |
     | security.md | 6   | 6         | 0          | ✓ PASS |
     ```

   - Рассчитайте общий статус:
     - **PASS**: Во всех чек-листах 0 невыполненных пунктов
     - **FAIL**: В одном или нескольких чек-листах есть невыполненные пункты

   - **Если какой-либо чек-лист не завершен**:
     - Отобразите таблицу с количеством невыполненных пунктов
     - **ОСТАНОВИТЕСЬ** и спросите: "Some checklists are incomplete. Do you want to proceed with implementation anyway? (yes/no)"
     - Ждите ответа пользователя перед продолжением
     - Если пользователь ответит "no", "wait" или "stop", остановите выполнение
     - Если пользователь ответит "yes", "proceed" или "continue", переходите к шагу 3

   - **Если все чек-листы завершены**:
     - Отобразите таблицу, показывающую, что все чек-листы пройдены
     - Автоматически переходите к шагу 3

3. Загрузите и проанализируйте контекст реализации:
   - **ОБЯЗАТЕЛЬНО**: Прочитайте `tasks.md` для получения полного списка задач и плана выполнения
   - **ОБЯЗАТЕЛЬНО**: Прочитайте `plan.md` для понимания стека технологий, архитектуры и структуры файлов
   - **ЕСЛИ ЕСТЬ**: Прочитайте `data-model.md` для понимания сущностей и связей
   - **ЕСЛИ ЕСТЬ**: Прочитайте `contracts/` для спецификаций API и требований к тестам
   - **ЕСЛИ ЕСТЬ**: Прочитайте `research.md` для технических решений и ограничений
   - **ЕСЛИ ЕСТЬ**: Прочитайте `quickstart.md` для сценариев интеграции

4. **Верификация настройки проекта**:
   - **ОБЯЗАТЕЛЬНО**: Создайте/проверьте файлы игнорирования на основе реальной настройки проекта:

   **Логика обнаружения и создания**:
   - Проверьте, выполняется ли следующая команда, чтобы определить, является ли репозиторий git-репозиторием (если да, создайте/проверьте `.gitignore`):

     ```sh
     git rev-parse --git-dir 2>/dev/null
     ```

   - Проверьте, существует ли `Dockerfile*` или упоминается Docker в `plan.md` → создайте/проверьте `.dockerignore`
   - Проверьте, существует ли `.eslintrc*` → создайте/проверьте `.eslintignore`
   - Проверьте, существует ли `eslint.config.*` → убедитесь, что записи `ignores` в конфиге покрывают требуемые шаблоны
   - Проверьте, существует ли `.prettierrc*` → создайте/проверьте `.prettierignore`
   - Проверьте, существует ли `.npmrc` или `package.json` → создайте/проверьте `.npmignore` (если планируется публикация)
   - Проверьте, существуют ли файлы terraform (`*.tf`) → создайте/проверьте `.terraformignore`
   - Проверьте, нужен ли `.helmignore` (присутствуют helm charts) → создайте/проверьте `.helmignore`

   **Если файл игнорирования уже существует**: Проверьте, содержит ли он основные шаблоны, добавьте только недостающие критические шаблоны
   **Если файл игнорирования отсутствует**: Создайте с полным набором шаблонов для обнаруженной технологии

   **Общие шаблоны по технологиям** (на основе стека в `plan.md`):
   - **Node.js/JavaScript/TypeScript**: `node_modules/`, `dist/`, `build/`, `*.log`, `.env*`
   - **Python**: `__pycache__/`, `*.pyc`, `.venv/`, `venv/`, `dist/`, `*.egg-info/`
   - **Java**: `target/`, `*.class`, `*.jar`, `.gradle/`, `build/`
   - **C#/.NET**: `bin/`, `obj/`, `*.user`, `*.suo`, `packages/`
   - **Go**: `*.exe`, `*.test`, `vendor/`, `*.out`
   - **Ruby**: `.bundle/`, `log/`, `tmp/`, `*.gem`, `vendor/bundle/`
   - **PHP**: `vendor/`, `*.log`, `*.cache`, `*.env`
   - **Rust**: `target/`, `debug/`, `release/`, `*.rs.bk`, `*.rlib`, `*.prof*`, `.idea/`, `*.log`, `.env*`
   - **Kotlin**: `build/`, `out/`, `.gradle/`, `.idea/`, `*.class`, `*.jar`, `*.iml`, `*.log`, `.env*`
   - **C++**: `build/`, `bin/`, `obj/`, `out/`, `*.o`, `*.so`, `*.a`, `*.exe`, `*.dll`, `.idea/`, `*.log`, `.env*`
   - **C**: `build/`, `bin/`, `obj/`, `out/`, `*.o`, `*.a`, `*.so`, `*.exe`, `Makefile`, `config.log`, `.idea/`, `*.log`, `.env*`
   - **Swift**: `.build/`, `DerivedData/`, `*.swiftpm/`, `Packages/`
   - **R**: `.Rproj.user/`, `.Rhistory`, `.RData`, `.Ruserdata`, `*.Rproj`, `packrat/`, `renv/`
   - **Универсальные**: `.DS_Store`, `Thumbs.db`, `*.tmp`, `*.swp`, `.vscode/`, `.idea/`

   **Специфичные для инструментов шаблоны**:
   - **Docker**: `node_modules/`, `.git/`, `Dockerfile*`, `.dockerignore`, `*.log*`, `.env*`, `coverage/`
   - **ESLint**: `node_modules/`, `dist/`, `build/`, `coverage/`, `*.min.js`
   - **Prettier**: `node_modules/`, `dist/`, `build/`, `coverage/`, `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`
   - **Terraform**: `.terraform/`, `*.tfstate*`, `*.tfvars`, `.terraform.lock.hcl`
   - **Kubernetes/k8s**: `*.secret.yaml`, `secrets/`, `.kube/`, `kubeconfig*`, `*.key`, `*.crt`

5. Разберите структуру `tasks.md` и извлеките:
   - **Фазы задач**: Настройка (Setup), Тесты (Tests), Ядро (Core), Интеграция (Integration), Полировка (Polish)
   - **Зависимости задач**: Правила последовательного и параллельного выполнения
   - **Детали задач**: ID, описание, пути к файлам, маркеры параллельности [P]
   - **Поток выполнения**: Порядок и требования к зависимостям

6. Выполните реализацию, следуя плану задач:
   - **Пофазное выполнение**: Завершите каждую фазу перед переходом к следующей
   - **Соблюдайте зависимости**: Запускайте последовательные задачи по порядку, параллельные задачи [P] могут выполняться вместе
   - **Следуйте подходу TDD**: Выполняйте задачи тестирования перед соответствующими задачами реализации
   - **Координация на основе файлов**: Задачи, затрагивающие одни и те же файлы, должны выполняться последовательно
   - **Контрольные точки валидации**: Проверяйте завершение каждой фазы перед продолжением

7. Правила выполнения реализации:
   - **Сначала настройка**: Инициализируйте структуру проекта, зависимости, конфигурацию
   - **Тесты перед кодом**: Если нужно написать тесты для контрактов, сущностей и сценариев интеграции
   - **Разработка ядра**: Реализуйте модели, сервисы, CLI-команды, эндпоинты
   - **Интеграционная работа**: Подключения к базам данных, middleware, логирование, внешние сервисы
   - **Полировка и валидация**: Юнит-тесты, оптимизация производительности, документация

8. Отслеживание прогресса и обработка ошибок:
   - Сообщайте о прогрессе после каждой выполненной задачи
   - Остановите выполнение, если любая непараллельная задача завершилась ошибкой
   - Для параллельных задач [P] продолжайте с успешными задачами, сообщите о неудавшихся
   - Предоставляйте четкие сообщения об ошибках с контекстом для отладки
   - Предлагайте следующие шаги, если реализация не может быть продолжена
   - **ВАЖНО** Для выполненных задач убедитесь, что отметили задачу как [X] в файле задач.

9. Валидация завершения:
   - Проверьте, что все требуемые задачи выполнены
   - Убедитесь, что реализованные функции соответствуют исходной спецификации
   - Проверьте, что тесты проходят и покрытие соответствует требованиям
   - Подтвердите, что реализация следует техническому плану
   - Сообщите финальный статус с кратким изложением выполненной работы

Примечание: Эта команда предполагает, что в `tasks.md` существует полная разбивка задач. Если задачи неполные или отсутствуют, предложите сначала запустить `/speckit.tasks` для регенерации списка задач.
